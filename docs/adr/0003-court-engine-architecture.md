# Архитектура движка: DSL → Scene-модель → рендерер → плеер

Движок схем разбит на слои с общей промежуточной моделью **Scene** в центре:

1. **DSL-парсер** (`court-dsl.js`) — текст на специальном DSL → Scene.
2. **Scene-модель** — нормализованные данные: `court`, `entities`, `steps` (группы одновременных действий с длительностями), `zones`. Единая промежуточная форма (IR).
3. **Рендерер** (`court-render-svg.js`) — `(scene, t) → SVG DOM`, чистая функция от сцены и позиции воспроизведения.
4. **Плеер** (`court-player.js`) — часы на `requestAnimationFrame`, playhead, управление (play/pause, шаг, перемотка, скорость).

Решение: **единственный источник Scene — DSL.** Рендерер и плеер знают только про Scene и не зависят от текстового синтаксиса.

Старый объект-формат диаграмм (`diagram: { shots, players, zones }` в `drills.js`) **не адаптируется, а заменяется**: все 24 диаграммы переписываются на DSL. Причина — старые схемы содержат ошибки (зеркальные лево/право, неверные направления), поэтому адаптер просто переносил бы баги вперёд. Чистая переавторизация на DSL заодно выверяет корректность.

## Consequences

- Адаптера `objectDiagram → Scene` нет — отвергнут сознательно: сохранять заведомо ошибочные данные смысла нет.
- 24 диаграммы переписываются на DSL «за один заход»; пока не переписаны — рендерятся старым `renderCourt`. Переход разовый, не постепенный.
- DSL — единственный вход Scene; его грамматику и рендерер можно менять независимо, пока контракт Scene стабилен.
- Длительности в DSL необязательны: парсер проставляет дефолты по типу действия; явное время — только для переопределения. Это решение про авторский комфорт, не про модель данных.
- Scene IR сохраняется как слой, несмотря на единственный вход: он отделяет авторинг от рендера и допускает несколько рендереров (см. [0002](./0002-svg-court-renderer.md)).
